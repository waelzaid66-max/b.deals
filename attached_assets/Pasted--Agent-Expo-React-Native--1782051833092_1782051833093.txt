# توصيات للـ Agent — إصلاح أيقونات Expo/React Native + ضمان الاستقرار والتوسّع

## أولًا: إصلاح مشكلة الأيقونات المكسورة (المربعات/Tofu)

### القاعدة الذهبية
لا تعرض الأيقونات عبر خطوط TTF (مثل @expo/vector-icons / createIconSet).
اعرضها كـ SVG مضمّن عبر `lucide-react-native` (تعتمد على react-native-svg).
السبب: خطوط الأيقونات تتعارض مع نسخ Expo Go على أندرويد فتظهر مربعات،
أما SVG فلا يحمّل أي خط → استحالة ظهور مربع فارغ على كل المنصات.

### خطوات التنفيذ
1. أنشئ ملف سجلّ واحد للأيقونات (مثلًا src/components/icons.ts) هو
   المكان الوحيد المسؤول عن الأيقونات في المشروع كله.
2. استورد الأيقونات من lucide-react-native فقط، واربط كل اسم أيقونة
   بمكوّن lucide في خريطة صريحة.
3. صدّر مكوّنات بنفس واجهة الاستخدام القديمة { name, size, color,
   style, onPress } حتى لا تضطر لتعديل مواقع الاستخدام.
4. لأي اسم غير معروف، اعرض أيقونة بديلة واضحة (fallback) بدل الانهيار
   أو المربع الفارغ.
5. في الـ root layout: حمّل خطوط النصوص فقط (Inter/Cairo)، ولا تحمّل
   أي خط أيقونات إطلاقًا.
6. لشريط التبويبات: استخدم نفس سجلّ SVG على كل المنصات. تجنّب
   expo-symbols (SF Symbols) و expo-glass-effect (NativeTabs) لأنها
   iOS-only وتتكسر في Expo Go. (يجوز إبقاء expo-blur للخلفية فقط.)
7. حافظ على سلامة الأنواع: لو محتاج أسماء الأيقونات كـ types، استخدم
   `typeof import("@expo/vector-icons")` كاستعلام نوع فقط (يُمحى وقت
   البناء) — بدون أي استيراد فعلي وقت التشغيل.

### اختبار حماية من التراجع (مهم جدًا)
أضف اختبار node:test يتحقق من:
- السجلّ مبني على SVG (لا createIconSet، لا عائلات خطوط مخصّصة).
- لا يوجد تحميل خط أيقونات في أي مكان.
- فقط ملف السجلّ يلمس @expo/vector-icons (وللأنواع فقط).
- كل اسم أيقونة مستخدم فعلًا مربوط بأيقونة حقيقية في السجلّ.
اربطه كـ script (مثل test:icons) وشغّله في الـ CI/قبل أي دمج.

## ثانيًا: ضمان الاستقرار (Stability)

1. مصدر واحد للحقيقة: أي مورد متكرر (أيقونات، ألوان، نصوص، صلاحيات)
   يكون له ملف/طبقة واحدة، وكل الباقي يستهلك منها. يمنع التكرار والانجراف.
2. ثبّت الإصدارات بدقّة: استخدم إصدارات Expo SDK الأساسية بدون ^ أو ~،
   ووحّد الإصدار في مكان واحد (في pnpm: overrides داخل
   pnpm-workspace.yaml — لأن pnpm v11+ يتجاهل حقل pnpm في package.json).
   وجود نسختين قابلتين للحل من نفس المكتبة = عودة الأعطال.
3. افشل بوضوح بدل الفشل الصامت: لا تستخدم بيانات وهمية أو fallback
   صامت يخفي الخطأ؛ اعرض الخطأ صراحةً أو أيقونة/حالة واضحة.
4. اختبارات حماية لكل إصلاح غير بديهي: أي bug تصلحه، اقفله باختبار
   يمنع رجوعه، ووثّق "لماذا" في تعليق قصير فوق الكود.
5. تحقّق على بيئة حقيقية: لقطة شاشة الويب لا تؤكد إصلاح tofu على
   أندرويد. الإثبات الحقيقي = جهاز فعلي + Metro نظيف + الاختبار.
6. عند ظهور المشكلة رغم الإصلاح: عاملها كمشكلة بيئة (كاش Expo Go
   قديم / Metro قديم / بناء native قديم) → امسح الكاش أو اعمل بناء نظيف.

## ثالثًا: التوسّع الآمن (Safe Scaling)

1. طبقات وعقود ثابتة (Contracts): عرّف عقد بيانات/واجهة ثابت بين
   الطبقات (Backend ↔️ Frontend) ولا تكسره؛ أي تغيير في الشكل = تحديث
   كل المستهلكين في نفس الوقت + bump لمفتاح الكاش لو موجود.
2. الأمان أولًا في كل سطح عام: أي endpoint أو شاشة عامة يجب أن تطبّق
   نفس شروط الصلاحية/الرؤية المركزية — لا تترك سطحًا واحدًا يتسرّب منه
   محتوى محظور. اجعل منطق الصلاحيات مركزيًا لا منسوخًا.
3. الأسرار عبر نظام الـ Secrets فقط: لا تكتب مفاتيح/توكنات في الكود
   أو الملفات؛ استخدم متغيّرات البيئة، وتحقّق من وجود Integration رسمي
   قبل طلب أي مفتاح من المستخدم.
4. التحقّق قبل الدمج: typecheck + lint + الاختبارات + بناء نظيف
   كبوابة إلزامية قبل أي دمج، حتى لا ينكسر شيء عند التوسّع.
5. وحدات صغيرة ومنفصلة: قسّم الملفات الكبيرة، افصل المكوّنات، وحافظ
   على بنية المجلدات؛ الكود المنظّم يتوسّع بأمان والكبير المتشابك ينكسر.
6. وثّق القرارات غير البديهية: أي قرار تصميمي مهم (لماذا SVG لا خطوط،
   لماذا إصدار مثبّت...) دوّنه باختصار حتى لا يعكسه أحد لاحقًا بالغلط.
