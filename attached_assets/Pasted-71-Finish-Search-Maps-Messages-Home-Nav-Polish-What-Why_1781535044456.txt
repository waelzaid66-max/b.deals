71 - Finish Search, Maps, Messages & Home/Nav Polish

What & Why

Complete and correctly finish the core BANCO Mobile discovery + communication surfaces as one cohesive batch, matching the agreed execution spec and the quality of the last published version (modern, dynamic — latest Instagram / Apple feel). The user audited the running app and found these surfaces half-finished or broken: the home header is cluttered with a useless search rectangle, the bottom navigation bar lost its glass look, the Search page opens nearly empty (and many agreed search features are incomplete or hidden), the Map does not display, and the Messages section is missing its Messenger features. Doing it as one batch avoids design/performance regressions across these connected surfaces.

Root causes confirmed during audit:

The map shows a "needs key" fallback because EXPO_PUBLIC_GOOGLE_MAPS_API_KEY is empty — the map code (price-pill pins, web + native) is otherwise ready.
The Messages chat list + thread exist but lack attachments, read indicators, typing indicator, and full real-time feel.
Done looks like

Home header is clean: the search-bar rectangle next to the logo is removed (it added no value and took space); search is reached only via the bottom-tab search icon.
AI assistant is a single chic icon in the header (no "AI" text), still opening the assistant.
Bottom navigation bar keeps its exact current icon order but gets the glass/blur (iOS Blur / Glassmorphism) applied to the whole bar, rendered uniformly on iOS, Android, and web, matching the last published design.
Search opens alive like Booking, never empty: an animated marketing banner/ticker (slogans "منزل أحلامك"، "عربية أحلامك"، "مصنع يدعم طموحك"، "وفّر واشترى من المنبع"، "استورد بنفسك" with imagery), main categories to browse, and smart suggestions show immediately on open. Auto-suggestions while typing are visible and working.
Search works dynamically and smartly like big booking apps: browse via the map, or search within a 7 km radius around the user, plus governorate / specific address / all-Egypt. All previously-agreed search system features that are currently incomplete or hidden are surfaced and finished (sort by cheapest/newest/nearest; ranking Verified → best price → high engagement → newest).
Rejected copy removed: "لاقي أصلك" is gone everywhere, replaced with clean professional Arabic that fits the app.
Maps actually render with the Google Maps key wired up: search-result vendors appear as price-pill pins on an interactive Google-Maps-style map, performant, wherever results have coordinates. Honest empty state only when there are genuinely no mappable results.
Messages feel like a real Messenger: clear conversation list + thread, image and PDF attachments, read indicators (sent/delivered/seen), typing indicator, tight real-time updates, simple Messenger-style UX, honest empty state when there are no conversations.
Whole experience is dynamic, modern, smart, uncluttered, and tidy — with page/data speed kept a priority.

Steps

Home header cleanup — Remove the search-bar rectangle from the home header (logo, AI entry, bell remain); search is reached via the bottom-tab search icon. Replace the AI icon+text pill with a single chic icon that still routes to the assistant. Keep RTL correct.
Glass navigation bar — Keep the current tab icon order unchanged. Apply the glass/blur (iOS Blur / Glassmorphism) to the entire bottom bar and ensure it renders uniformly on iOS, Android, and web (cross-platform blur, not iOS-only modules), matching the last published look. Follow the existing compatibility note in _layout.tsx.
Search entry experience — Replace the empty pre-query state with a Booking-like rich state on open: an animated auto-rotating marketing banner/ticker pairing the slogans above with imagery from attached_assets, main categories to browse, trending and recent searches, and smart suggestions. Ensure auto-suggestions while typing are visible and working.
Search copy cleanup — Remove search.introTitle ("لاقي أصلك") and any other awkward wording; replace with clean professional Arabic, keeping English equivalents in sync.
Dynamic search + geo (Near Me 7 km) — Make search behave like big booking apps: browse via map or search within a 7 km radius around the user (plus governorate / specific address / all Egypt). Request location permission up front. Add user-selectable sort (cheapest / newest / nearest) and apply the agreed ranking order. Keep it fast.
Audit & finish hidden search features — Deeply verify the existing search system and surface/finish anything agreed but incomplete or not showing (filters, save-search/alerts, autocomplete), so the section is fully functional and visible, not partially wired.
Maps render & display — Wire up the Google Maps API key so the map renders instead of the "needs key" fallback. Show search-result vendors as price-pill pins on an interactive Google-Maps-style map, performant, wherever results have coordinates, in both ListingMap.web.tsx (web) and ListingMap.tsx (native). CRITICAL: the key is a secret/env value — coordinate with the user to provide/enable it; never hardcode it.
Results presentation — Tighten result card layout, spacing, imagery, and ensure each result's data is fully and cleanly shown in both list and map views, with finished empty/error states.
Messages — Messenger-grade completion — Bring chat up to spec (§10): verified conversation list + thread, image and PDF attachments, read indicators (sent/delivered/seen), typing indicator, tighter real-time updates, simple Messenger-style UX, honest empty state. Do not fabricate stored chat history.
Polish pass — Verify the full flow on mobile (home → search → map → results → listing → chat) is dynamic, modern, tidy, consistent: no leftover rectangle, no "AI" text, no rejected copy, glass nav bar uniform, map renders, chat complete.

Source banner imagery from attached_assets (recent .jpeg files and IMG_*.PNG). Do not invent stock that misrepresents listings or fabricate inventory.
Cross-platform blur via expo-blur (already used for the iOS tab background); make it uniform across platforms.
Keep page/data speed a priority (spec §2.3) — no lag from the new map/search/banner.
