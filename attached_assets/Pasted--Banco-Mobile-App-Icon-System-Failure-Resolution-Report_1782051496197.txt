دةBanco Mobile App — Icon System Failure & Resolution Report

1) Executive Summary

واجه التطبيق مشكلة حرجة في عرض الأيقونات على أجهزة Android (خصوصًا عبر Expo Go)، حيث ظهرت جميع الأيقونات كمربعات (tofu boxes □)، مما أثر بشكل مباشر على واجهة المستخدم وتجربة الاستخدام.

تم حل المشكلة بشكل جذري عبر استبدال نظام الأيقونات بالكامل من Font-based إلى SVG-based، مما أدى إلى استقرار العرض وإزالة الاعتماد على تحميل الخطوط.

⸻

2) Root Cause Analysis (تحليل السبب الجذري)

النظام السابق:

* يعتمد على: @expo/vector-icons
* يستخدم: Font-based icons (Feather / Ionicons / MCI)

المشكلة:

* الأيقونات تعتمد على تحميل font files وقت التشغيل
* في بيئة Expo (Android):
    * تحميل الفونت غير مضمون
    * أحيانًا لا يتم تسجيل الفونت (font registration fails)

النتيجة:

* كل icon يتحول إلى:
    □ (tofu box)

السبب الجذري الحقيقي:

Failure in runtime font loading → glyphs not resolved → fallback tofu rendering

⸻

3) Why Previous Fix Attempts Failed

المحاولات السابقة كانت:

* إعادة تحميل الكاش
* إعادة تشغيل Expo
* تعديل font preload

❌ كلها فشلت لأن:

المشكلة ليست في الكاش، بل في طبيعة النظام نفسه (font dependency)

⸻

4) Implemented Solution (الحل المنفذ)

القرار الأساسي:

إزالة الاعتماد على fonts بالكامل

⸻

4.1 Migration to SVG Icons

تم استبدال:

* Font Icons → SVG Icons

باستخدام:

* lucide-react-native
* react-native-svg

قبل:

<Feather name="home" />

بعد:

<House />

4.2 Architectural Layer (icons.ts)

تم إنشاء طبقة وسيطة:

* ملف مركزي: icons.ts
* يحافظ على نفس interface القديمة:

keyof typeof Feather.glyphMap

الفائدة:

* عدم تعديل باقي المشروع
* استبدال backend فقط (Font → SVG)
* Decoupling كامل بين UI و icon system

⸻

4.3 Removing Font Dependency

تم:

* حذف preload الخاص بالأيقونات من _layout.tsx
* الإبقاء فقط على:
    * Cairo
    * Inter

النتيجة:

التطبيق لم يعد يعتمد على أي icon fonts

⸻

4.4 Full Icon Mapping

تم:

* حصر كل الأيقونات المستخدمة (~40+ icon)
* مطابقتها مع lucide equivalents
* معالجة edge cases:
    * MCI (finance)
    * arrow variations
    * single-character icons (مثل X)

4.5 Testing & Validation

تم إضافة:

* Unit tests للتحقق من:
    * وجود كل icon
    * صحة mapping
    * استخدام SVG

تم إصلاح:

* regex issue في matching (support single-character components)

بعد تطبيق الحل:

المشكلة كانت:

System-level dependency (fonts)

* Fix bug

الحل كان:

Replace entire rendering mechanism

8) Architectural Impact

قبل:

* Tight coupling مع font system
* Runtime dependency

بعد:

* Decoupled icon system
* Static rendering (SVG)
* Predictable behavior

9) Lessons Learned

1. Font-based icons غير موثوقة في Expo Android
2. SVG هو الحل الأكثر استقرارًا
3. Abstraction layer (icons.ts) مهم جدًا
4. إصلاح الأعراض (cache / reload) لا يحل المشاكل الجذرية
5. الحل الصحيح أحيانًا يكون إزالة النظام بالكامل

* الانتقال إلى:
    * Performance optimization
    * UX polishing
    * Feature completion

Final Statement

تم حل مشكلة الأيقونات بشكل جذري عبر استبدال نظام يعتمد على runtime fonts بنظام يعتمد على static SVG rendering، مما أدى إلى استقرار كامل في عرض الأيقونات عبر جميع الأجهزة.

كانت كل الأيقونات في التطبيق تظهر كمربعات فارغة (□) أو علامة X
(الحرف المعروف بـ ".notdef" أو "tofu") على أندرويد داخل Expo Go،
في كل مكان: شريط التبويبات السفلي، صفوف القوائم، أزرار الكروت... إلخ.
## 2) السبب الجذري (Root Cause)
المشكلة كانت في طريقة عرض الأيقونات عبر خطوط TTF من مكتبة
`@expo/vector-icons`:
- كل مجموعة أيقونات (Feather / Ionicons / MaterialCommunityIcons)
  تُحمّل ملف خطها TTF تحت اسم عائلة خط ثابت
  (feather / ionicons / material-community).
- لكن تطبيق Expo Go نفسه يكون قد سجّل بالفعل نسخته الخاصة من نفس
  الخطوط تحت نفس أسماء العائلات (وغالبًا بإصدار مختلف).
- النظام يتجاهل إعادة تسجيل عائلة خط محجوزة مسبقًا، فتصبح خريطة
  الرموز (glyph map) في كود التطبيق تشير إلى الخط الخطأ → كل
  الأيقونات تتكسر وتظهر كمربعات.
ملاحظة مهمة: محاولات الحل بإعادة تسمية العائلات إلى أسماء فريدة
(banco-*) أو تثبيت إصدار واحد بدقّة كانت تنجح "نظريًا" فقط، وظلّت
هشّة جدًا مع كل تحديث كاش لـ Expo Go أو ترقية SDK. أي أن حل الخطوط
نفسه كان "مسمومًا" وغير موثوق.
## 3) الحل النهائي المعتمد
التخلي تمامًا عن خطوط الأيقونات (TTF)، وعرض كل الأيقونات كـ
رسومات SVG مضمّنة عبر مكتبة `lucide-react-native` (التي ترسم
باستخدام `react-native-svg` الموجودة أصلًا كاعتمادية).
النتيجة: لا يوجد أي خط أيقونات يتم تحميله إطلاقًا، وبالتالي ظهور
مربع فارغ (missing-glyph) أصبح مستحيلًا هيكليًا على كل المنصات
(أندرويد / iOS / ويب / Expo Go / dev / production).
## 4) تفاصيل التنفيذ
### أ) سجلّ أيقونات مملوك للتطبيق
الملف: `artifacts/banco-mobile/src/components/icons.ts`
- هو المكان الوحيد المسؤول عن الأيقونات في التطبيق كله.
- يستورد الأيقونات من `lucide-react-native` فقط.
- يُصدّر نفس المكوّنات القديمة بالأسماء نفسها:
  `Feather` / `Ionicons` / `MaterialCommunityIcons`
  بنفس الواجهة تمامًا: { name, size, color, style, onPress }
  → لذلك مواقع الاستخدام في التطبيق لم تتغيّر إطلاقًا.
- كل مكوّن مدعوم بخريطة صريحة: اسم الأيقونة → مكوّن lucide.
- أي اسم غير معروف يسقط على أيقونة بديلة واضحة (CircleAlert)،
  فلا يحدث انهيار ولا يظهر مربع فارغ أبدًا.
### ب) خدعة الحفاظ على سلامة الأنواع (Type-safety)
- المستهلكون يكتبون أسماء الأيقونات كـ
  `keyof typeof Feather.glyphMap`.
- للإبقاء على هذا يعمل دون استيراد فعلي لـ `@expo/vector-icons`
  وقت التشغيل، يستخدم الملف استعلام نوع فقط:
  `type VectorIcons = typeof import("@expo/vector-icons")`
  وهذا يُمحى وقت البناء (build) ولا يضيف أي اعتمادية وقت التشغيل
  ولا يحمّل أي خط → فلا يمكن أن يُعيد المشكلة.
- القاعدة: ملف icons.ts فقط هو المسموح له بلمس
  `@expo/vector-icons`، وللأنواع فقط.
### ج) عدم تحميل أي خط أيقونات عند الإقلاع
الملف: `artifacts/banco-mobile/app/_layout.tsx`
- استدعاء `useFonts(...)` يحمّل خطوط النصوص فقط
  (Inter للإنجليزية و Cairo للعربية).
- لا يوجد أي خط أيقونات (لا bancoIconFonts ولا غيره)، مع تعليق
  صريح يشرح السبب لمنع أي تراجع مستقبلي.
### د) شريط التبويبات وعناصر التنقل
- شريط التبويبات السفلي يعرض أيقوناته عبر سجلّ SVG على كل المنصات.
- ممنوع استخدام `expo-symbols` (SF Symbols) أو
  `expo-glass-effect` (NativeTabs)، لأنها وحدات iOS أصلية غير
  موجودة في Expo Go وتُظهر رمزًا مكسورًا.
- يجوز إبقاء `expo-blur` فقط لخلفية iOS الشفافة.
## 5) اختبار الحماية من التراجع (Regression Test)
